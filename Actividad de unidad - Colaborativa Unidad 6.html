SQL vs NoSQL en Café Andino S.A.S.

Comparativo práctico para una empresa con tres sucursales (Pereira, Manizales y Armenia): creación de dos bases de datos, análisis de ventajas y desventajas, procedimiento, conclusiones y bibliografía.

Presentación

Este blog presenta un análisis claro y aplicado sobre el uso de bases de datos SQL y NoSQL en una empresa colombiana ficticia, Café Andino S.A.S., dedicada a la producción y comercialización de café artesanal con tres sucursales a nivel nacional. Se explica el procedimiento de creación de ambas bases de datos, se incluyen fragmentos de código y se discuten ventajas y desventajas para fundamentar la elección más conveniente.

Introducción

En el entorno empresarial, elegir el tipo de base de datos impacta la manera en que se almacenan, protegen y consultan los datos. Las bases de datos relacionales (SQL) priorizan la estructura, las reglas y la consistencia; mientras que las NoSQL se caracterizan por esquemas flexibles y escalabilidad horizontal, útiles para datos cambiantes o de alto volumen. Con base en los contenidos de la Unidad 6, se identifican las características funcionales de cada enfoque y se aplican en un caso de uso realista para Café Andino S.A.S.

1) Base de datos SQL (MySQL)

Se implementó un modelo relacional en MySQL con integridad referencial para gestionar ventas, inventario y sucursales.

2) Base de datos NoSQL (MongoDB – documental)

Para comparar, se creó una base en MongoDB Atlas con la colección ventas que almacena documentos JSON. Este modelo facilita la flexibilidad y el análisis.

3) Comparativo general

CriterioSQL (MySQL)NoSQL (MongoDB)
EstructuraRelacional, esquema fijoDocumental, esquema flexible
ConsistenciaAlta (ACID)Eventual/ajustable
EscalabilidadPrincipalmente verticalHorizontal (sharding, réplicas)
RendimientoExcelente en OLTPMuy alto en lecturas masivas/anidadas
Modelo de datosNormalización y FKsDocumentos JSON anidados

Conclusión técnica: SQL ofrece orden y garantías transaccionales; NoSQL destaca por flexibilidad y escalado. Un enfoque híbrido aprovecha lo mejor de ambos.

Conclusiones

  • Ambos enfoques son válidos; la elección depende del objetivo y tipo de datos.
  • Para Café Andino S.A.S., se recomienda un esquema híbrido: MySQL para operaciones y MongoDB para analítica de clientes.
  • El diseño del modelo y los índices es clave para el rendimiento.
